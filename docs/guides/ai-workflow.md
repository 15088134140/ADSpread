# AI 驱动开发工作流实践指南

> 📚 基于 105 Agent 深度研究结果的最佳实践总结

---

## 核心认知

### 1. AI 的能力边界

| AI 能做的 | AI 做不好的 |
|----------|------------|
| 按规范编写代码 | 模糊需求的创造性设计 |
| 编写单元测试 | 跨系统的架构权衡 |
| 定位已知模式的 Bug | 复杂的根因分析 |
| 代码格式和风格检查 | 业务逻辑的深层理解 |
| 多视角 Code Review | 发布决策的最终责任 |

### 2. Hermes 的独特价值

当前 Claude Code 生态中**没有**成熟的"主控调度"产品，Hermes 是本项目的原创设计：

- 统一的任务入口和质量把关点
- 管理多个 Claude Code 实例的上下文隔离
- 决策何时调用何种 Superpowers 技能
- 处理 Agent 间的冲突和意见分歧

## 日常开发工作流

### 标准流程（适用于 90% 的普通功能）

```
1. 需求理解
   └─ Hermes: 阅读 PRD/设计文档，识别边界和验收标准

2. 任务拆解
   └─ Hermes: 拆分为 1-2 个文件的原子任务，定义验收标准

3. 编码实现
   └─ Claude Code: 实现功能 + 单元测试

4. 自验
   └─ Claude Code:
      - TypeScript 类型检查
      - ESLint 检查
      - 运行单元测试
      - 本地手动验证

5. 提交
   └─ Hermes: 最终检查 → 提交代码
```

**何时不需要 Superpowers**: 纯 UI 改动、工具函数、简单的 CRUD 接口

---

### 复杂功能流程（核心业务模块）

```
1. 需求理解 + 方案设计
   └─ Hermes
      └─ [可选] 调用 Superpowers brainstorming
         └─ 多方案对比 + 风险识别

2. 任务拆解
   └─ Hermes: 拆分为可并行子任务，定义接口契约

3. 并行开发（如启动子 Agent）
   ├─ Claude Code (子 Agent A): 模块 A
   ├─ Claude Code (子 Agent B): 模块 B
   └─ Claude Code (子 Agent C): 模块 C

4. Code Review 门禁
   └─ Hermes 调用 Superpowers code-review
      ├─ 正确性审查
      ├─ 安全性审查
      ├─ 性能审查
      └─ 可维护性审查

5. 修复意见处理
   └─ Claude Code: 按 Review 意见修复

6. 集成和最终验收
   └─ Hermes: 集成 + 端到端验证
```

**触发 Superpowers 的标准**:
- ✅ 涉及认证、权限、支付等安全敏感逻辑
- ✅ 算法复杂度 O(n²) 以上
- ✅ 跨 3 个以上模块的改动
- ✅ 首次使用新技术/新框架
- ✅ 核心业务流程

## Superpowers 技能使用指南

### `brainstorming` - 头脑风暴

**使用场景**:
- 技术选型决策前（数据库、框架、第三方服务）
- 架构设计方案评审
- 复杂 Bug 的根因分析
- 需求不明确时的多维度拆解

**调用方式**:
```
/skill brainstorming

主题: [明确的问题，如"用户认证方案选型对比"]
约束: [列出技术栈、性能、成本等约束]
```

**输出期望**:
- 3-5 个可行方案
- 每个方案的 Pros/Cons
- 风险清单和应对措施
- 推荐方案和理由

---

### `code-review` - 代码审查

**使用场景**:
- 核心模块开发完成后
- PR 合并前的最终把关
- 重构后的质量验证

**审查维度**:
| 维度 | 检查项 |
|------|--------|
| **正确性** | 边界条件、异常处理、逻辑漏洞 |
| **安全性** | SQL 注入、XSS、认证授权、数据泄露 |
| **性能** | N+1 查询、内存泄漏、算法复杂度 |
| **可维护性** | 命名、注释、复杂度、代码重复 |
| **一致性** | 是否符合项目规范和架构设计 |

**输出格式**:
```
Code Review 报告
├─ Critical: [数量] 个必须修复的阻塞性问题
├─ Major: [数量] 个建议修复的重要问题
├─ Minor: [数量] 个可以后续优化的问题
└─ Nit: [数量] 个风格和小细节问题
```

---

### `subagent-driven-development` - 多 Agent 协作

**使用场景**:
- 预估超过 5 个文件的大型功能
- 需要多领域知识的复杂任务
- 时间紧、可以并行开发的任务

**三角色模型**:
```
Implementer (实现者)
    ↓ 产出代码
Reviewer (审查者)
    ↓ 独立审查代码
QA (测试者)
    ↓ 编写测试并验证
```

> 💡 关键机制: 三个角色完全隔离上下文，避免思维定势和错误传播。

---

### `verify` - 功能验证

**使用场景**:
- P0 Bug 修复后
- 核心业务流程变更后
- 发布前的最终验证

**验证内容**:
- 端到端测试场景设计
- 回归测试用例覆盖
- 边界条件验证
- 异常流程验证

---

## 代码评审标准速查

### ✅ 通过标准

- 所有 TypeScript 类型检查通过
- 无 ESLint 错误
- 单元测试覆盖率 ≥ 80%
- Code Review 中 **无 Critical 级别问题**
- Major 级别问题 ≤ 2 个，且有后续优化计划

### ⚠️ 需要警惕的代码味道

| 代码味道 | 说明 |
|----------|------|
| 深层嵌套 | if/for 嵌套超过 3 层 |
| 巨型函数 | 单函数超过 100 行 |
| 魔法数字 | 未定义常量的数字直接使用 |
| 注释缺失 | 复杂算法无说明 |
| 重复代码 | 3 处以上完全相同的逻辑 |
| 过度抽象 | 为了设计模式而设计模式 |

## 常见问题 FAQ

### Q: 每次 Code Review 都需要调用 Superpowers 吗？

**A**: 不需要。建议:
- 简单 CRUD: 跳过，Claude Code 自验即可
- 普通业务逻辑: 选择性调用，10 次提交调用 1-2 次
- 核心模块: **必须**调用

### Q: Superpowers 的 Code Review 意见和 Claude Code 冲突怎么办？

**A**: Hermes 作为最终仲裁者:
1. 对比双方的理由和证据
2. 优先考虑安全性、正确性
3. 仍无法判断时，引入人类决策者

### Q: 子 Agent 协作一定会更快吗？

**A**: **不一定**。多 Agent 协作:
- ✅ **提升质量**: 多视角审查，减少错误
- ⚠️ **可能变慢**: 通信开销、意见整合、冲突解决
- 📊 经验法则: 小于 3 个文件的任务，不要用子 Agent

### Q: 如何避免 AI 幻觉？

**A**: 三道防线:
1. **Claude Code 自验**: 运行测试，实际执行代码
2. **Superpowers 独立审查**: 不继承上下文，新鲜视角
3. **Hermes 最终把关**: 关键决策点要求 AI 提供证据链

## 反模式 - 不要这样做

### ❌ 反模式 1: 过度依赖 AI

> "把需求丢给 AI，等着代码自己写好。"

**问题**: AI 对业务的理解永远不如人，产出的代码可能偏离需求。

**正确做法**: 明确的验收标准 + 人最终确认功能正确性。

---

### ❌ 反模式 2: 教条化流程

> "每个提交都必须经过 Superpowers Code Review，否则不合并。"

**问题**: 流程 overhead 远大于收益，降低开发效率。

**正确做法**: 按需调用，风险与成本权衡。

---

### ❌ 反模式 3: 让 AI 做决策但人不负责任

> "AI 说这么做的，出问题找它。"

**问题**: AI 不承担法律和业务责任，最终必须有人负责。

**正确做法**: AI 提供建议，人做最终决策并承担责任。
