# 信发系统管理端与服务端 MVP 设计规格

**日期**: 2026-06-25  
**范围**: 管理端前端与对应服务端 MVP 纵向闭环  
**参考文档**:

- [产品需求文档](../../requirements/信发系统_产品需求文档.md)
- [技术设计文档](../../architecture/信发系统_技术设计文档.md)

---

## 1. 背景与目标

本设计用于落实信发系统产品需求文档中管理端前端和对应服务端的 MVP 功能。Android 客户端不在本次实现范围内，但服务端会提供未来 Android 可调用的设备节目拉取接口。

当前代码库已经具备 Monorepo 骨架、NestJS 后端、Vue3 管理端、Prisma schema 和基础共享类型，但业务模块与页面尚未完整落地，且现有 schema、共享类型、响应格式与 PRD/技术设计文档存在差异。因此本次工作采用“文档对齐优先 + 主业务纵向闭环”的方式推进。

MVP 完成后，应能在浏览器中完成：

登录 → 新建门店 → 新建设备 → 上传素材 → 审核素材 → 创建并发布节目 → 创建发布计划 → 点击推送并看到推送结果 → 通过设备编码接口拉取当前节目。

本项目以以下两份文档作为标准参照：

- `docs/requirements/信发系统_产品需求文档.md`

5. 不纳入 MVP 的完整功能要在本规格中明确列为后续阶段，不做伪实现。

## 3. MVP 范围与边界

### 3.1 纳入 MVP 的功能

#### 登录

- 后端真实管理员登录接口。
- JWT 鉴权。
- 前端登录页、token 保存、路由守卫。
- 左侧菜单使用静态菜单，不做动态 RBAC 过滤。

#### 门店管理

- 列表、搜索、行业分类筛选、分页。
- 新增、编辑、删除。
- 行业分类必填。
- 有设备的门店不可删除。
- 状态启用/禁用。

#### 设备管理

- 列表、搜索、按门店筛选、分页。
- 新增、编辑、删除。
- 所属门店允许为空。
- 屏幕方向、分辨率、分屏类型、备注、状态。
- 分辨率必填。
- 状态只表示启用/禁用；在线状态通过最后活跃时间展示。
- 分屏类型根据横竖屏做基础校验。

#### 素材管理

- 真实文件上传到后端本地目录。
- 支持图片 JPG/JPEG/PNG/GIF，视频 MP4/AVI/MOV。
- 单文件 100MB 限制。
- 上传后默认为待审核。
- 列表支持名称、类型、审核状态筛选。
- 图片预览，视频展示 URL/文件信息。
- 审核通过、审核驳回。
- 驳回原因必填且至少 10 个字符。
- 只有审核通过素材可用于节目。

#### 节目制作

- 节目列表、搜索、状态筛选。
- 新建、编辑、删除。
- 保存草稿、保存并发布。
- 配置屏幕方向：横屏、竖屏、任意。
- 配置分屏类型：1、2、3、3-1、4、任意。
- 根据实际分屏类型生成区域。
- 每个区域选择一个或多个已审核通过素材。
- 每个素材展示时长 MVP 默认 5 秒，并在数据结构中保留 `duration` 字段。

#### 发布管理

- 发布计划列表、搜索、状态筛选、分页。
- 新建、编辑、删除。
- 只能选择已发布节目。
- 只支持按门店发布，多选门店。
- 生效开始时间、可选结束时间。
- 播放周期周一至周日多选，至少一天。
- 启用/停用。
- 立即推送、批量推送：MVP 只计算目标设备数量、更新 `lastPushedAt`、记录推送结果，不建立真实 WebSocket 连接。

#### 设备拉取节目接口

- 通过设备编码查询当前应播放的节目。
- 筛选规则遵循 PRD：设备存在、门店启用、设备启用、发布计划启用、当前时间命中、当前星期命中。
- 多个计划命中时返回最新创建的计划对应节目。

### 3.2 不纳入本次 MVP 的功能

- Android 客户端。
- 真实 Socket.io 设备长连接和 ack。
- 完整 RBAC 管理：管理员账号、角色管理、菜单管理 CRUD。
- 动态菜单权限过滤。
- 完整多语言切换。
- 完整操作日志页面。
- Excel 批量导入设备。
- APK 升级、统计报表、素材标签、节目优先级、时段轮播等后续规划。

## 4. 方案选择

### 4.1 备选方案

#### 方案 1：主业务纵向闭环

按“登录 → 基础数据 → 素材审核 → 节目制作 → 发布计划 → 设备拉取接口”的业务链路推进。

优点：最快形成可演示、可验收的完整闭环；风险可控。  
缺点：RBAC、多语言、完整操作日志不会在 MVP 内完整实现。

#### 方案 2：后端 API 优先

先完成数据库、DTO、服务、控制器、Swagger/接口契约和测试，再集中做前端。

优点：接口稳定，前端对接更顺。  
缺点：短期看不到完整后台操作效果，验收体验较弱。

#### 方案 3：前端 Demo 优先

先完成管理端页面、表单、表格、交互，用 mock 或轻量接口撑起流程，再补后端真实逻辑。

优点：最快看到界面和流程。  
缺点：后续替换真实接口时返工风险高。

### 4.2 决策

采用方案 1：主业务纵向闭环。

原因是当前仓库更接近业务骨架，后端模块和前端页面都尚未完整落地。纵向闭环可以避免“只完成一端不能用”，并能用一条真实主流程持续验证每个阶段。

## 5. 后端架构与模块边界

后端继续使用 `apps/backend` 的 NestJS + Prisma 架构，按业务领域拆模块。

### 5.1 AuthModule

- `POST /api/auth/login`
- 返回 JWT、管理员信息。
- MVP 初始化/种子数据提供默认管理员。
- 不实现完整 refresh token 流程，先按 24 小时 access token 满足登录需求。

### 5.2 StoreModule

- 门店 CRUD。
- 行业分类选项接口。
- 删除前检查门店下设备数量。

### 5.3 DeviceModule

- 设备 CRUD。
- 门店可为空。
- 分辨率选项接口。
- 分屏类型按屏幕方向校验。

### 5.4 MaterialModule

- 文件上传、列表、删除。
- 审核通过/驳回。
- 可用素材列表：仅返回审核通过素材，供节目制作使用。

### 5.5 ProgramModule

- 节目 CRUD。
- 草稿/发布。
- 保存区域素材配置。
- 保存时校验引用素材必须为审核通过。

### 5.6 PublishModule

- 发布计划 CRUD。
- 启用/停用。
- 单个推送/批量推送记录。
- 推送时计算目标门店下设备数并更新 `lastPushedAt`。

### 5.7 Device Read API

面向未来 Android 的只读接口：

- `GET /api/device/program?deviceCode=DEVICE001`

本次只实现拉取节目，不实现心跳和 WebSocket。

### 5.8 PrismaModule

作为统一数据库访问入口。不引入额外 repository 抽象，避免过度设计。

## 6. 前端架构与模块边界

管理端继续使用 `apps/admin` 的 Vue3 + Element Plus + Pinia。

页面模块：

- `views/auth/`：登录页。
- `views/store/StoreList.vue`：门店管理。
- `views/device/DeviceList.vue`：设备管理。
- `views/material/MaterialList.vue`：素材管理与审核弹窗。
- `views/program/ProgramList.vue`：节目制作。
- `views/publish/PublishList.vue`：发布管理。
- `layouts/MainLayout.vue`：固定侧边栏菜单，MVP 不做动态菜单。

API 层新增或补齐：

- `src/api/auth.ts`
- `src/api/store.ts`
- `src/api/device.ts`
- `src/api/material.ts`
- `src/api/program.ts`
- `src/api/publish.ts`

请求统一走 `src/utils/request.ts`，避免组件中裸用 axios。

边界约定：

- 页面组件负责表格、表单、弹窗交互。
- API 调用封装在 `src/api/*`。
- 除登录用户外，业务列表状态先留在页面组件内。
- 不做通用 CRUD 生成器。

## 7. 数据模型与接口契约

### 7.1 统一响应格式

本项目是新项目，不需要兼容旧返回码。全项目统一采用技术设计文档中的响应格式：

```ts
{
  code: 0,
  message: 'success',
  data: T,
  timestamp: number
}
```

要求：

- 后端成功响应固定 `code: 0`。
- `timestamp` 统一为毫秒时间戳 number。
- 前端只接受 `code === 0` 为成功。
- 共享类型中 `ApiResponse.timestamp` 为 number。
- 文档、注释、示例如有不同，统一改为 `code: 0`。

### 7.2 Prisma 对齐方向

当前 `schema.prisma` 与文档存在差异，MVP 实施中按文档统一：

1. 设备 `storeId` 改为可空。
2. 删除门店不级联删除设备；有设备门店不可删除。
3. 设备分屏类型按文档语义统一为 `1 / 2 / 3 / 3-1 / 4`。
4. 设备分辨率必填。
5. 设备 `status` 只表示启用/禁用；在线状态通过 `lastActiveAt` 表示。
6. 发布计划 `status` 只表示启用/停用。
7. 发布计划 `playDays` 必填，至少一天。
8. 节目支持“任意屏幕方向”和“任意分屏类型”。
9. 素材文件大小按技术设计使用 BIGINT。
10. 节目状态数值统一为技术设计：`0` 草稿，`1` 已发布。
11. 全项目响应格式统一为成功 `code: 0`、`timestamp: number`。

### 7.3 管理员与登录

保留 `Admin`、`Role`、`Menu` 模型，但 MVP 只使用 `Admin` 登录。

- 默认管理员通过 seed 或初始化脚本创建。
- 密码使用 bcrypt 哈希。
- MVP 不做完整 RBAC，但如果 PRD/技术设计要求管理员必须关联角色，则实现时应创建默认超级管理员角色并保持数据约束一致。

### 7.4 门店

字段：

- `name`
- `code`
- `industryCategory`
- `address`
- `contactPerson`
- `contactPhone`
- `status`
- `remark`
- `createdAt`
- `updatedAt`

行业分类以技术设计文档为准，API 层对外返回文档定义值和展示标签。

### 7.5 设备

字段：

- `storeId` 可空。
- `name`
- `code`
- `screenOrientation`
- `screenResolution` 必填。
- `splitType`
- `remark`
- `status`: `1` 启用，`0` 禁用。
- `lastActiveAt`

屏幕方向和分屏类型校验：

- 竖屏：1、2、3。
- 横屏：1、2、3、3-1、4。

### 7.6 素材

字段：

- `auditStatus` 默认待审核。
- `auditUserId`
- `auditTime`
- `auditReason`
- `createdBy`
- `fileSize` 使用 BIGINT。

文件存储：

- 后端保存到 `apps/backend/uploads/materials/yyyy/mm/dd/uuid.ext` 或环境变量配置目录。
- 返回 `fileUrl`，前端用于预览/打开。
- MVP 做扩展名、MIME、大小校验；魔数校验作为后续安全增强。

### 7.7 节目

`layoutConfig` 作为 JSON 保存：

```json
{
  "regions": [
    {
      "regionId": "region1",
      "materials": [{ "materialId": 1, "duration": 5 }]
    }
  ]
}
```

区域坐标不强制存入数据库。设备拉取当前节目接口返回时，根据屏幕方向和分屏类型补齐坐标。

状态：

- `0` 草稿。
- `1` 已发布。

### 7.8 发布计划

字段：

- `programId`
- `name`
- `targetStoreIds`
- `startTime`
- `endTime`
- `playDays`
- `status`
- `lastPushedAt`

状态：

- `1` 启用。
- `0` 停用。

`playDays` 必填，至少一天。

## 8. 前端页面与交互设计

MVP 前端以“筛选栏 + 表格 + 分页 + 弹窗表单”为统一页面模式，遵循 Element Plus 风格。

### 8.1 登录与布局

登录页：

- 调用真实 `POST /api/auth/login`。
- 登录成功后保存 token 和 `userInfo`。
- 跳转到 `/dashboard`。
- 失败时使用后端错误消息。
- 移除 mock 语义。

布局与菜单：

- 保留静态侧边栏菜单。
- 菜单项覆盖仪表盘、门店管理、设备管理、素材管理、节目制作、发布管理。
- 系统管理菜单 MVP 推荐隐藏，避免误认为 RBAC/日志已完成。
- 退出登录清理 token 并跳转登录页。

### 8.2 门店管理页面

筛选：

- 关键词：门店名称/编码。
- 行业分类。
- 状态。

表格字段：

- 门店名称、门店编码、行业分类、地址、联系人、联系电话、状态、设备数量、创建时间、操作。

表单校验：

- 门店名称必填。
- 门店编码必填。
- 行业分类必填。
- 联系电话使用宽松校验，不强制中国手机号格式。
- 删除时二次确认。

### 8.3 设备管理页面

筛选：

- 关键词：设备名称/编码。
- 所属门店。
- 状态。

表格字段：

- 设备名称、设备编码、所属门店、屏幕方向、屏幕分辨率、分屏类型、备注、状态、最后活跃时间、创建时间、操作。

表单交互：

- 所属门店可为空。
- 屏幕方向为横屏/竖屏。
- 分屏类型随屏幕方向动态过滤。
- 切换屏幕方向后，如果当前分屏类型不合法，则清空并要求重新选择。
- 分辨率使用常见选项并支持自定义输入。

### 8.4 素材管理页面

筛选：

- 关键词：素材名称。
- 类型：图片/视频。
- 审核状态。

表格字段：

- 素材名称、类型、文件大小、文件扩展名、审核状态、审核人、审核时间、驳回原因、上传时间、操作。

上传：

- 使用 Element Plus Upload。
- 上传接口：`POST /api/materials/upload`。
- 限制单文件 100MB。
- 前端先做快速校验，后端强制校验。
- 上传成功后刷新列表。

审核：

- 审核弹窗展示素材名称、类型、大小、URL。
- 图片可预览。
- 视频显示 URL/文件名，不强制内嵌播放。
- 支持审核通过和审核驳回。
- 驳回原因必填且至少 10 个字符。

### 8.5 节目制作页面

筛选：

- 关键词：节目名称。
- 状态：草稿/已发布。

表格字段：

- 节目名称、屏幕方向、分屏类型、状态、发布时间、创建时间、操作。

新建/编辑节目：

- 节目名称必填。
- 适配屏幕方向：横屏、竖屏、任意。
- 适配分屏类型：1、2、3、3-1、4、任意。
- 根据实际分屏类型生成 region1/region2 等区域。
- 每个区域可选择多个已审核通过素材。
- 每个素材默认 duration = 5 秒。
- 支持保存草稿和保存并发布。

校验：

- 保存草稿：节目名称、屏幕方向、分屏类型必填。
- 保存并发布：每个实际区域至少有一个素材。
- 后端再次校验所有素材必须审核通过。

### 8.6 发布管理页面

筛选：

- 关键词：发布计划名称/节目名称。
- 状态：启用/停用。

表格字段：

- 发布计划名称、节目名称、目标门店、开始时间、结束时间、播放周期、状态、最后推送时间、创建时间、操作。

新建/编辑发布计划：

- 发布计划名称必填。
- 节目必选，只显示已发布节目。
- 目标门店多选，至少一个。
- 生效开始时间必填。
- 生效结束时间可空；如填写必须晚于开始时间。
- 播放周期多选，至少一天。
- 状态：启用/停用。

推送：

- 单个推送返回目标设备数量和推送记录结果。
- 批量推送支持表格多选。
- MVP 提示文案明确为“已记录推送并更新推送时间”，不暗示真实设备已收到。

### 8.7 前端通用约定

- 日期格式统一使用 `YYYY-MM-DD HH:mm:ss`。
- 页面内状态局部维护，不为列表查询引入全局 store。
- API 调用集中在 `src/api/*`。
- 表单规则靠近页面定义，暂不抽成通用规则库。
- 所有后端返回错误直接展示 `message`。
- 新增页面文字先使用中文硬编码；后续多语言阶段统一迁移到 i18n。

## 9. 后端业务规则与错误处理

### 9.1 认证与鉴权

- 管理端业务接口默认需要 JWT。
- 登录接口和设备只读接口例外：`/api/auth/login`、`/api/device/program`。
- MVP 不按菜单/角色做接口权限校验。

### 9.2 门店规则

- 门店编码唯一。
- 行业分类必填。
- 删除门店前检查是否存在关联设备。
- 禁用门店后，设备拉取节目接口不返回该门店发布计划。

### 9.3 设备规则

- 设备编码唯一。
- 所属门店可为空。
- 如传入 `storeId`，必须存在对应门店。
- 屏幕方向和分屏类型必须匹配。
- 无门店设备调用节目接口时返回 `data: null`。

### 9.4 素材规则

- 上传文件大小不超过 100MB。
- 上传后默认待审核。
- 审核通过写入审核人和审核时间，并清空旧驳回原因。
- 审核驳回要求原因至少 10 个字符。
- 删除素材时，如果素材已被节目引用，拒绝删除。

### 9.5 节目规则

- 节目名称必填。
- 屏幕方向和分屏类型必须符合文档。
- 保存草稿允许区域素材为空。
- 发布节目要求每个实际区域至少一个素材。
- 发布节目要求所有素材存在且审核通过。
- 已被发布计划引用的节目不能删除。

### 9.6 发布计划规则

- 只能选择已发布节目。
- 目标门店至少一个。
- 所有目标门店必须存在。
- 开始时间必填。
- 结束时间可空；如填写必须晚于开始时间。
- 播放周期至少一天。
- 推送根据目标门店查询启用设备，更新 `lastPushedAt`，返回目标设备数量。

### 9.7 设备拉取当前节目规则

接口：`GET /api/device/program?deviceCode=DEVICE001`

筛选流程：

1. 根据设备编码查设备。
2. 设备不存在：返回业务错误 `404xx`。
3. 设备禁用：返回 `data: null`。
4. 设备无门店：返回 `data: null`。
5. 门店禁用：返回 `data: null`。
6. 查询包含该门店 ID 的启用发布计划。
7. 筛选当前时间在 `[startTime, endTime]` 范围内；`endTime` 为空表示长期有效。
8. 筛选当前星期命中 `playDays`。
9. 按 `createdAt` 倒序取最新计划。
10. 返回节目、分屏、区域、素材 URL 和 duration。

### 9.8 错误格式

所有错误统一：

```json
{
  "code": 40001,
  "message": "参数错误",
  "data": null,
  "timestamp": 1782360000000
}
```

错误码约定：

- `400xx`：参数/业务规则错误。
- `401xx`：未登录、token 无效。
- `403xx`：无权限。
- `404xx`：资源不存在。
- `500xx`：服务端错误。

## 10. 文件上传安全

MVP 强制：

- 文件大小限制。
- 扩展名白名单。
- MIME 白名单。
- 文件名 UUID 化。
- 不使用用户上传原文件名作为存储名。
- 静态文件路径只暴露 uploads 下的素材访问 URL。
- 日志中不记录 token、密码。

魔数校验和图片二次渲染作为后续安全增强，不阻塞 MVP。

## 12. 测试策略

本 MVP 的测试策略采用“务实增强”方案：后端覆盖核心业务规则和关键 API 集成路径；前端不额外引入测试框架，但必须通过类型/构建校验，并形成可执行的手工验收清单。测试目标不是追求形式上的覆盖率，而是确保主业务闭环、文档规则和高风险校验真实可靠。

### 12.1 测试目标

测试必须覆盖以下风险点：

1. 统一响应格式是否始终为 `code: 0`、`timestamp: number`。
2. 登录鉴权是否真实生效。
3. 门店、设备、素材、节目、发布计划的核心业务规则是否由后端强制执行。
4. 素材上传、审核和节目引用之间的约束是否可靠。
5. 发布计划当前节目筛选规则是否符合 PRD。
6. 前端核心页面是否能完成主流程，且表单校验与后端规则一致。
7. MVP 范围内代码、共享类型、PRD 和技术设计文档是否一致。

### 12.2 后端单元测试

后端优先补 Service 层单元测试。Service 单测使用 mock PrismaService，聚焦业务分支，不依赖真实数据库。

建议覆盖：

- `AuthService`
  - 用户名不存在时登录失败。
  - 管理员禁用时登录失败。
  - 密码错误时登录失败。
  - 登录成功返回 token 和用户信息。
- `StoreService`
  - 门店编码唯一。
  - 行业分类必填。
  - 有设备的门店不可删除。
  - 禁用门店后不影响设备数据本身。
- `DeviceService`
  - 门店可为空。
  - 传入门店 ID 时必须存在。
  - 分辨率必填。
  - 竖屏只允许 1/2/3 分屏。
  - 横屏允许 1/2/3/3-1/4 分屏。
  - 设备状态只接受启用/禁用。
- `MaterialService`
  - 上传默认待审核。
  - 文件类型白名单校验。
  - 文件大小限制。
  - 驳回原因必填且至少 10 个字符。
  - 审核通过后清空旧驳回原因。
  - 已被节目引用的素材不能删除。
- `ProgramService`
  - 草稿可保存空区域素材。
  - 发布节目时每个实际区域至少一个素材。
  - 未审核通过素材不能用于发布。
  - 节目状态数值为 `0` 草稿、`1` 已发布。
  - 已关联发布计划的节目不能删除。
- `PublishService`
  - 只能选择已发布节目创建发布计划。
  - 目标门店至少一个。
  - 结束时间必须晚于开始时间。
  - 播放周期至少一天。
  - 推送时只统计启用设备并更新 `lastPushedAt`。
  - 当前节目筛选优先最新创建的有效发布计划。

### 12.3 后端 API 集成测试

除 Service 单测外，MVP 至少增加关键 API 集成测试。集成测试使用 Nest TestingModule + supertest，验证路由、DTO 校验、鉴权、统一响应格式和数据库读写路径。

建议覆盖最小 API 集：

1. `POST /api/auth/login`
   - 正确账号密码返回 `code: 0`、token、用户信息。
   - 错误密码返回统一错误结构。
2. `GET /api/stores`
   - 未登录访问被拒绝。
   - 登录后返回分页结构。
3. `POST /api/stores` + `DELETE /api/stores/:id`
   - 可创建门店。
   - 有设备时删除门店失败。
4. `POST /api/devices`
   - 可创建无门店设备。
   - 非法屏幕方向/分屏组合失败。
5. `POST /api/materials/upload`
   - 合法文件上传成功，状态为待审核。
   - 非白名单文件失败。
6. `POST /api/materials/:id/approve` / `reject`
   - 审核通过成功。
   - 驳回原因过短失败。
7. `POST /api/programs` / `POST /api/programs/:id/publish`
   - 草稿保存成功。
   - 使用未审核素材发布失败。
   - 使用审核通过素材发布成功。
8. `POST /api/publish` / `POST /api/publish/:id/push`
   - 已发布节目可创建发布计划。
   - 推送返回目标设备数量并更新最后推送时间。
9. `GET /api/device/program?deviceCode=DEVICE001`
   - 命中当前时间和播放周期时返回节目。
   - 设备禁用、门店禁用、无有效计划时返回 `data: null`。

如果实施时尚未配置独立测试数据库，应在实施计划中明确补齐测试数据库方案。集成测试不得污染开发数据；每个测试用例需要创建并清理自己的测试数据，或在测试前后重置测试库。

## 13. 风险与应对

### 风险 1：当前后端模块引用但实际文件缺失

`app.module.ts` 已引用多个模块，但初步检索未看到业务模块文件。

应对：

- 实施计划第一步增加现状核查。
- 若确实缺失，按 NestJS 业务模块新建。
- 保持模块结构简单：controller/service/dto。

### 风险 2：文档之间存在局部冲突

PRD 与技术设计文档整体一致，但细节可能有差异，例如枚举值命名、状态码数值、目录命名。

应对：

- 不绕过文档。
- 先修正文档，再实现。
- 在实施报告中列出所有文档同步点。

### 风险 3：素材文件访问路径

本地上传后需要让前端能访问文件。

应对：

- 后端启用 uploads 静态目录。
- `fileUrl` 返回 `/uploads/materials/...`。
- 前端预览使用该 URL。
- 生产环境 Nginx 静态服务作为后续部署配置，不阻塞本地 MVP。

### 风险 4：节目编辑 UI 复杂度

分屏区域 + 多素材选择可能导致页面较复杂。

应对：

- MVP 不做可视化拖拽。
- 使用按区域分组的表单。
- 坐标预览和拖拽编排后续再做。

### 风险 5：完整 RBAC、多语言、真实 WebSocket 未纳入 MVP

PRD 完整要求包含这些能力，但本次 MVP 暂不全部实现。

应对：

- 在规格文档中明确“完整产品目标”和“MVP 实现范围”的区别。
- 不做伪实现。
- 后续阶段按 PRD/技术设计继续补齐。
