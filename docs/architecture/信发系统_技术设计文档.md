## 1. 系统概述

### 1.1 项目背景

信发系统是一套面向连锁门店的广告信息发布系统，由Web管理后台和安卓客户端两大部分组成，实现：

- 集中管理多门店、多设备的广告内容
- 支持多种分屏布局展示不同广告内容
- 定时生效和远程更新广告节目
- 安卓客户端7x24小时稳定运行
- v1.3新增：门店行业分类管理、素材审核流程

### 1.2 业务目标

| 目标       | 说明                             |
| ---------- | -------------------------------- |
| 统一管理   | 集中管理1000+门店，5000+设备     |
| 分屏展示   | 支持横屏/竖屏多种分屏布局        |
| 实时推送   | 广告更新秒级生效                 |
| 稳定运行   | 客户端长时间稳定运行，开机自启动 |
| 多语言支持 | 日语、中文简体、英语三种语言     |
| 行业分类   | 支持8种行业分类管理门店          |
| 素材审核   | 完善的素材审核机制，确保内容合规 |

## 2. 整体架构设计

### 2.1 架构分层

```
┌─────────────────────────────────────────────────────────┐
│                     客户端层                              │
├─────────────┬───────────────────┬───────────────────────┤
│ 管理后台Web │  安卓展示客户端   │   设备设置APP         │
│ Vue3 + TS   │  Kotlin原生       │                       │
└─────────────┴───────────────────┴───────────────────────┘
                              ↓ HTTPS / WebSocket
┌─────────────────────────────────────────────────────────┐
│                  Nginx 反向代理层                        │
│            静态资源 + API代理 + SSL终止 + 限流              │
└─────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────┐
│                  NestJS 应用层                           │
├─────────────┬─────────────┬─────────────┬─────────────┤
│   认证模块   │  门店设备   │  素材节目   │  发布分发   │
│   RBAC      │   管理      │   管理      │   管理      │
│             │ 行业分类     │  审核流程    │  批量推送    │
├─────────────┼─────────────┼─────────────┼─────────────┤
│  系统管理   │  操作日志   │  多语言     │ Socket.io   │
│  (菜单/角色)│  全链路审计  │   i18n      │  推送服务   │
└─────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────┐
│                    数据存储层                             │
├─────────────────────┬───────────────────────────────────┤
│      MySQL 8.0      │           Redis 7                 │
│   业务数据持久化     │    缓存 + Session + 限流         │
└─────────────────────┴───────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────┐
│                    文件存储层                             │
├─────────────────────────────────────────────────────────┤
│                    本地文件系统                           │
│              /data/adspread/materials/              │
│                按日期分目录存储                       │
└─────────────────────────────────────────────────────────┘
```

### 2.2 架构设计原则

1. **模块化优先**：NestJS每个功能模块独立，接口清晰，便于未来拆分微服务
2. **类型安全**：全栈TypeScript，前后端类型定义共享
3. **高内聚低耦合**：业务逻辑在Service层，Controller仅做HTTP适配
4. **可观测性**：全链路日志、请求ID追踪、性能监控
5. **渐进式扩展**：单体起步，预留未来拆分微服务的架构边界
6. **设计一致性**：统一设计系统，确保UI/UX一致性

### 4.2 核心表详细设计

#### 4.2.1 门店表 (stores)

| 字段名            | 类型                                                                                                 | 约束                                          | 说明                                                  |
| ----------------- | ---------------------------------------------------------------------------------------------------- | --------------------------------------------- | ----------------------------------------------------- |
| id                | INT                                                                                                  | PRIMARY KEY, AUTO_INCREMENT                   | 主键ID                                                |
| name              | VARCHAR(100)                                                                                         | NOT NULL                                      | 门店名称                                              |
| code              | VARCHAR(50)                                                                                          | NOT NULL, UNIQUE                              | 门店编码，唯一标识                                    |
| industry_category | ENUM('catering', 'retail', 'beauty', 'hospitality', 'education', 'automotive', 'localLife', 'other') | NOT NULL                                      | 行业分类：餐饮/零售/美妆/酒旅/教育/汽车/本地生活/其他 |
| address           | VARCHAR(255)                                                                                         | NULL                                          | 门店地址                                              |
| contact_person    | VARCHAR(50)                                                                                          | NULL                                          | 联系人                                                |
| contact_phone     | VARCHAR(20)                                                                                          | NULL                                          | 联系电话                                              |
| status            | TINYINT                                                                                              | NOT NULL, DEFAULT 1                           | 状态：0禁用 1启用                                     |
| created_at        | DATETIME                                                                                             | NOT NULL, DEFAULT CURRENT_TIMESTAMP           | 创建时间                                              |
| updated_at        | DATETIME                                                                                             | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间                                              |

**索引：** `idx_code` (UNIQUE), `idx_status`, `idx_industry_category`

**业务规则：**

- 门店编码必须全局唯一
- 行业分类为必填字段，创建时必须选择
- 门店下有设备时不允许删除
- 禁用门店下的所有设备无法接收新广告

#### 4.2.2 设备表 (devices)

| 字段名             | 类型                          | 约束                                          | 说明                                 |
| ------------------ | ----------------------------- | --------------------------------------------- | ------------------------------------ |
| id                 | INT                           | PRIMARY KEY, AUTO_INCREMENT                   | 主键ID                               |
| store_id           | INT                           | NULL, FOREIGN KEY                             | 所属门店ID，可为空                   |
| name               | VARCHAR(100)                  | NOT NULL                                      | 设备名称                             |
| code               | VARCHAR(50)                   | NOT NULL, UNIQUE                              | 设备编码，客户端标识                 |
| screen_orientation | ENUM('landscape', 'portrait') | NOT NULL                                      | 屏幕方向：landscape横屏 portrait竖屏 |
| screen_resolution  | VARCHAR(20)                   | NOT NULL                                      | 分辨率：1920x1080等，支持自定义      |
| split_type         | VARCHAR(10)                   | NOT NULL                                      | 分屏类型：1/2/3/3-1/4                |
| remark             | TEXT                          | NULL                                          | 备注信息（故障记录等）               |
| status             | TINYINT                       | NOT NULL, DEFAULT 1                           | 状态：0禁用 1启用                    |
| last_active_at     | DATETIME                      | NULL                                          | 最后活跃时间                         |
| created_at         | DATETIME                      | NOT NULL, DEFAULT CURRENT_TIMESTAMP           | 创建时间                             |
| updated_at         | DATETIME                      | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间                             |

**索引：** `idx_code` (UNIQUE), `idx_store_id`, `idx_status`, `idx_last_active`

**分屏类型说明：**

| 屏幕方向 | 分屏类型 | 说明                           |
| -------- | -------- | ------------------------------ |
| 竖屏     | 1        | 全屏一个区域                   |
| 竖屏     | 2        | 上下两个相等区域               |
| 竖屏     | 3        | 上中下三个相等区域             |
| 横屏     | 1        | 全屏一个区域                   |
| 横屏     | 2        | 左右两个相等区域               |
| 横屏     | 3        | 左中右三个相等区域             |
| 横屏     | 3-1      | 左侧占1/2，右侧上下两个各占1/4 |
| 横屏     | 4        | 田字格四个相等区域             |

#### 4.2.3 素材表 (materials)

| 字段名         | 类型                   | 约束                                          | 说明                                  |
| -------------- | ---------------------- | --------------------------------------------- | ------------------------------------- |
| id             | INT                    | PRIMARY KEY, AUTO_INCREMENT                   | 主键ID                                |
| name           | VARCHAR(100)           | NOT NULL                                      | 素材名称                              |
| type           | ENUM('image', 'video') | NOT NULL                                      | 类型：image图片 video视频             |
| file_url       | VARCHAR(255)           | NOT NULL                                      | 文件访问URL                           |
| file_size      | BIGINT                 | NOT NULL                                      | 文件大小（字节）                      |
| file_extension | VARCHAR(10)            | NOT NULL                                      | 文件扩展名                            |
| width          | INT                    | NULL                                          | 图片宽度（像素）                      |
| height         | INT                    | NULL                                          | 图片高度（像素）                      |
| duration       | INT                    | NULL                                          | 视频时长（秒）                        |
| audit_status   | TINYINT                | NOT NULL, DEFAULT 0                           | 审核状态：0待审核 1审核通过 2审核驳回 |
| audit_user_id  | INT                    | NULL, FOREIGN KEY                             | 审核人ID                              |
| audit_time     | DATETIME               | NULL                                          | 审核时间                              |
| audit_reason   | VARCHAR(500)           | NULL                                          | 驳回原因（驳回时必填，最小10字符）    |
| created_by     | INT                    | NOT NULL, FOREIGN KEY                         | 上传人ID                              |
| created_at     | DATETIME               | NOT NULL, DEFAULT CURRENT_TIMESTAMP           | 创建时间                              |
| updated_at     | DATETIME               | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间                              |

**索引：** `idx_type`, `idx_audit_status`, `idx_created_by`, `idx_created_at`

**审核状态说明：**

| 状态值 | 状态名称 | 颜色标识     | 说明                     |
| ------ | -------- | ------------ | ------------------------ |
| 0      | 待审核   | 橙色 #fa8c16 | 素材刚上传，等待审核     |
| 1      | 审核通过 | 绿色 #52c41a | 审核通过，可在节目中使用 |
| 2      | 审核驳回 | 红色 #f5222d | 审核不通过，不可使用     |

**业务规则：**

- 素材上传后默认状态为"待审核"
- 只有"审核通过"的素材才能在节目中被选择
- 审核驳回时必须填写驳回原因（至少10个字符）
- 已审核通过的素材可以重新审核
- 支持的图片格式：JPG、PNG、GIF
- 支持的视频格式：MP4、AVI、MOV
- 单文件最大限制：100MB

#### 4.2.4 节目表 (programs)

| 字段名             | 类型                                 | 约束                                          | 说明                                 |
| ------------------ | ------------------------------------ | --------------------------------------------- | ------------------------------------ |
| id                 | INT                                  | PRIMARY KEY, AUTO_INCREMENT                   | 主键ID                               |
| name               | VARCHAR(100)                         | NOT NULL                                      | 节目名称                             |
| screen_orientation | ENUM('landscape', 'portrait', 'any') | NOT NULL                                      | 适配屏幕方向：landscape/portrait/any |
| split_type         | VARCHAR(10)                          | NOT NULL                                      | 适配分屏类型：1/2/3/3-1/4/any        |
| layout_config      | JSON                                 | NOT NULL                                      | 分屏区域配置                         |
| status             | TINYINT                              | NOT NULL, DEFAULT 0                           | 状态：0草稿 1已发布                  |
| created_by         | INT                                  | NOT NULL, FOREIGN KEY                         | 创建人ID                             |
| published_at       | DATETIME                             | NULL                                          | 发布时间                             |
| created_at         | DATETIME                             | NOT NULL, DEFAULT CURRENT_TIMESTAMP           | 创建时间                             |
| updated_at         | DATETIME                             | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间                             |

**layout_config JSON结构：**

```json
{
  "regions": [
    {
      "regionId": "region1",
      "x": 0,
      "y": 0,
      "width": 0.5,
      "height": 1,
      "materials": [
        { "materialId": 1, "duration": 10 },
        { "materialId": 2, "duration": 15 }
      ]
    }
  ]
}
```

**业务规则：**

- 节目中只能引用审核通过的素材
- 已关联到发布计划的节目不能删除
- 草稿状态的节目可以编辑，已发布的节目需重新发布后生效

#### 4.2.5 发布计划表 (publish_plans)

| 字段名           | 类型         | 约束                                          | 说明                                 |
| ---------------- | ------------ | --------------------------------------------- | ------------------------------------ |
| id               | INT          | PRIMARY KEY, AUTO_INCREMENT                   | 主键ID                               |
| program_id       | INT          | NOT NULL, FOREIGN KEY                         | 关联节目ID（只能关联已发布的节目）   |
| name             | VARCHAR(100) | NOT NULL                                      | 发布计划名称                         |
| target_store_ids | JSON         | NOT NULL                                      | 目标门店ID数组 [1,2,3]               |
| start_time       | DATETIME     | NOT NULL                                      | 生效开始时间                         |
| end_time         | DATETIME     | NULL                                          | 生效结束时间，null表示永久           |
| play_days        | VARCHAR(20)  | NOT NULL                                      | 播放周期：1,2,3,4,5,6,7 (周一到周日) |
| status           | TINYINT      | NOT NULL, DEFAULT 1                           | 状态：0停用 1启用                    |
| last_pushed_at   | DATETIME     | NULL                                          | 最后推送时间                         |
| created_by       | INT          | NOT NULL, FOREIGN KEY                         | 创建人ID                             |
| created_at       | DATETIME     | NOT NULL, DEFAULT CURRENT_TIMESTAMP           | 创建时间                             |
| updated_at       | DATETIME     | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间                             |

**索引：** `idx_program_id`, `idx_status`, `idx_created_by`, `idx_start_time`

**业务规则：**

- 只能选择已发布的节目创建发布计划
- 按门店发布，发布后该门店下所有设备都会接收到广告
- 播放周期为多选，广告只在选中的星期几播放
- 支持手动立即推送，推送后设备立即刷新节目

#### 4.2.6 角色表 (roles)

| status     | TINYINT      | NOT NULL, DEFAULT 1                           | 状态：0禁用 1启用                                                                                        |
| menu_ids   | JSON         | NULL                                          | 授权菜单 ID 数组（如 `[1,2,3,7]`），勾选到按钮级（type=3）；超级管理员角色留空，运行时识别为全部启用菜单 |
| created_at | DATETIME     | NOT NULL, DEFAULT CURRENT_TIMESTAMP           | 创建时间                                                                                                 |
| updated_at | DATETIME     | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间                                                                                                 |

**索引：** `idx_name` (UNIQUE), `idx_status`

**业务规则：**

- 角色名称全局唯一。
- 有关联管理员的角色不可删除。
- 名称约定为 `超级管理员` 的角色不可删除、不可修改其 `menu_ids`，运行时自动放行所有接口。
- `menu_ids` 为按钮级菜单 ID 数组，派生权限码集合时直接取这些菜单的 `permission` 字段去重，不递归子菜单。
- 禁用角色后，其关联的所有管理员账号都无法登录系统。

#### 4.2.7 菜单表 (menus)

| 字段名     | 类型         | 约束                                          | 说明                                                                     |
| ---------- | ------------ | --------------------------------------------- | ------------------------------------------------------------------------ |
| id         | INT          | PRIMARY KEY, AUTO_INCREMENT                   | 主键ID                                                                   |
| parent_id  | INT          | NULL                                          | 父菜单ID，NULL表示顶级菜单；自关联，删除有子菜单的节点时拒绝             |
| name       | VARCHAR(50)  | NOT NULL                                      | 菜单名称（同时作为多语言 key）                                           |
| path       | VARCHAR(200) | NULL                                          | 路由路径；目录与按钮节点为空，菜单节点必填                               |
| component  | VARCHAR(200) | NULL                                          | 前端页面组件路径（预留，当前实现静态注册路由）                           |
| icon       | VARCHAR(50)  | NULL                                          | Element Plus 图标组件名                                                  |
| sort       | INT          | NOT NULL, DEFAULT 0                           | 排序号，同级数字越小越靠前                                               |
| type       | TINYINT      | NOT NULL, DEFAULT 1                           | 节点类型：1目录/2菜单/3按钮                                              |
| permission | VARCHAR(100) | NULL                                          | 权限码（如 `device:list`）；目录节点为空，菜单节点同时代表"进入页面"权限 |
| status     | TINYINT      | NOT NULL, DEFAULT 1                           | 状态：0禁用 1启用                                                        |
| created_at | DATETIME     | NOT NULL, DEFAULT CURRENT_TIMESTAMP           | 创建时间                                                                 |
| updated_at | DATETIME     | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间                                                                 |

**索引：** `idx_parent_id`, `idx_sort`, `idx_status`

**菜单层级与三级建模说明：**

- 采用目录（type=1）→ 菜单（type=2）→ 按钮（type=3）三级结构。
- 目录节点用于侧边栏分组（如"系统管理"），无 `path`/`permission`。
- 菜单节点对应一个前端路由，`permission` 同时控制路由进入与列表接口。
- 按钮节点无 `path`，仅用于按钮显隐与接口授权（如"新增""删除"）。
- 一级菜单展示在左侧导航栏，二级菜单/按钮收纳在一级菜单下。
- 删除菜单时若有子菜单则拒绝；角色通过 `menu_ids` 勾选按钮级节点派生权限码集合。

#### 4.2.8 角色菜单权限授权（Role.menuIds）

> **实现说明（v1.3 第二阶段同步）**：早期设计采用 `role_menu_permissions` 中间表（`role_id` + `menu_id` + `permission_type`）。实际实现采用更轻量的 `Role.menuIds: number[]`（JSON 数组，存于 `roles` 表，见 §4.2.6），与 seed 数据结构与 PRD §7.1"菜单权限ID列表"描述一致。本节保留中间表设计的历史说明供参考，但**以 `Role.menuIds` 方案为准**。

**实际实现方案（`Role.menuIds`）：**

| 字段             | 类型 | 说明                                                                                                                    |
| ---------------- | ---- | ----------------------------------------------------------------------------------------------------------------------- |
| `roles.menu_ids` | JSON | 授权菜单 ID 数组，勾选到按钮级（type=3 节点）；空数组或 null 表示无授权（超级管理员角色例外，运行时识别为全部启用菜单） |

- 权限码集合 `S` 派生：`menu.findMany({ where: { id: { in: menuIds }, status: 1 } })` 取 `permission` 去重，不递归子菜单。
- 超级管理员角色（名称约定 `超级管理员`）`menu_ids` 留空，运行时自动放行所有接口，且不可删除、不可修改其 `menu_ids`。
- 分配权限接口 `PUT /api/admin/roles/:id/menus`（权限码 `role:assign`）整体覆盖 `menu_ids`。

**为什么不用中间表（设计取舍）：**

| 维度                       | 中间表 `role_menu_permissions`               | `Role.menuIds`（JSON，采纳）                                                           |
| -------------------------- | -------------------------------------------- | -------------------------------------------------------------------------------------- |
| 写入                       | 角色授权变更需 diff 后批量增删多行           | 一次 update 覆盖整个数组                                                               |
| 读取                       | 角色权限需 join 两张表                       | 单行读取即得全部授权 ID                                                                |
| 按钮级权限                 | 需 `permission_type` 列区分 view/edit/delete | 菜单表本身用 type=3 按钮节点 + `permission` 字段表达，`menuIds` 勾选按钮节点即得权限码 |
| 反查"某菜单被哪些角色授权" | SQL 友好                                     | 需 JSON 查询（MySQL `JSON_CONTAINS`）                                                  |
| Seed 一致性                | 需额外维护中间表 seed                        | 与角色 upsert 同步落库                                                                 |
| 适配规模                   | 千级角色 × 百级菜单                          | 当前规模读写性能足够                                                                   |

- 当前规模（角色数 < 100，菜单数 < 200）下 JSON 方案读写性能足够，且 seed 与角色 upsert 天然一致，简化实现。
- 反查需求可通过 `JSON_CONTAINS(menu_ids, CAST(? AS JSON))` 满足。
- 若未来角色/菜单规模显著增长或需要复杂权限类型（如字段级），可回退为中间表设计；届时同步迁移 `menu_ids` 数据并修订本节。

**记录方式：**

- 控制器写操作通过 `@OperationLog(operation, targetType?)` 声明，由全局 `OperationLogInterceptor` 在响应后/异常后异步写入，异常时透传原异常不吞错、不阻塞响应。
- 登录成功、登出在 `AuthService` 显式调用 `OperationLogService.create` 落库（拦截器覆盖不到非控制器调用）。

**需要记录的操作类型：**

- 发布计划：新建、编辑、启用/停用、删除、推送、批量推送
- 管理员：新增、编辑、删除、重置密码
- 角色：新增、编辑、删除、分配权限
- 菜单：新增、编辑、删除

### 4.3 数据库设计规范

1. **字段命名**：统一使用 snake_case 下划线命名
2. **时间戳**：所有表统一 created_at / updated_at 自动维护
3. **软删除**：不使用物理删除，重要数据用 status 标记删除
4. **JSON字段**：半结构化数据使用JSON，便于扩展
5. **冗余字段**：日志表等查询频繁场景适度冗余
6. **索引优化**：所有外键、筛选字段、排序字段建立合适索引
7. **字符集**：统一使用 utf8mb4 字符集，支持emoji
8. **引擎**：统一使用 InnoDB 引擎，支持事务和外键

### 5.4 国际化支持

**默认语言**：日语（遵 PRD §4.12）。前端 `main.ts` 初始化 `locale: localStorage.getItem('locale') || 'ja'`，`fallbackLocale: 'ja'`；用户切换后写入 `localStorage` 记忆选择，下次登录保持。Element Plus locale 通过 `ElConfigProvider` 响应式联动。

**支持语言**：日语（ja）、中文简体（zh-CN）、英语（en）。翻译文件按模块拆分：common、menu、store、device、material、program、publish、system、validation。

**请求头：**

```
Accept-Language: ja / zh-CN / en
```

前端 `utils/request.ts` 在每个请求头携带 `Accept-Language`（来自 `appStore.locale`），为后端未来国际化预留。

**国际化范围（v1.3 第二阶段实现状态）：**

- 前端全量 i18n：所有菜单、按钮、表单 label、表格列标题、校验消息、`ElMessage`/`ElMessageBox` 文案均已迁移为 `t()` key，无硬编码中文。

### 5.5 第二阶段实现要点汇总（RBAC / 操作日志 / 多语言 / Excel 导入）

本节集中说明 v1.3 第二阶段落地的四项核心机制，供快速理解全栈权限与审计闭环。

#### 5.5.1 全局权限守卫 `PermissionGuard`（严格兜底）

- 全局 `APP_GUARD` 注册，在 `JwtAuthGuard` 认证之后执行授权决策（按顺序短路）：
  1. `@Public()` → 放行（登录、`GET /api/device/program`）。
  2. `request.user` 不存在 → 401。
  3. 角色名 === `超级管理员` → 放行所有接口。
  4. `@AuthenticatedOnly()` → 放行（仅需登录：`/api/auth/me`/`/menus`/`/logout`/`/change-password`）。
  5. `@RequirePermission(code)` 命中且 `code ∈ 用户有效权限码集合 S` → 放行；否则 403。
  6. **未声明任何标记的受保护接口 → 一律 403 拒绝**（严格兜底，强制所有受保护接口显式声明权限码，避免漏挂导致越权）。
- 权限码集合 `S` 计算：取 `role.menuIds` 对应的 `status=1` 菜单的 `permission` 去重，请求内即时计算（本轮不引入 Redis 缓存）。
- 接口↔权限码映射见设计规格 `docs/superpowers/specs/2026-06-26-adspread-phase2-rbac-audit-i18n-excel-design.md` §4.4.2；HTTP method 偏差见 §5.3.10。

#### 5.5.2 操作日志全链路审计 `OperationLogInterceptor`

- 全局 `APP_INTERCEPTOR` 注册 + `@OperationLog(operation, targetType?)` 方法级装饰器。
- 自动记录写操作：adminId/roleId/menuId/username/operation/method/params（密码脱敏）/time（耗时 ms）/ip/userAgent/status/errorMsg（见 §4.2.10）。
- 异常时透传原异常不吞错，异步写入不阻塞响应。
- 登录/登出在 `AuthService` 显式落库（拦截器覆盖不到非控制器调用）。

#### 5.5.3 动态菜单与权限路由

- 登录后前端调 `GET /api/auth/menus` 获取当前用户可见菜单树，`stores/permission.ts` 扁平化取权限码填 `Set<string>`，供按钮级 `hasPermission(code)` 与路由守卫使用。
- **路由静态注册 + 守卫拦截**：所有路由在构建期静态注册（`meta.permission` 标注所需权限码），`router.beforeEach` 校验目标路由 `meta.permission` 是否在用户权限码集合内，未授权跳转。侧边栏按 `menuTree` 动态渲染（目录用 `el-sub-menu`，菜单用 `el-menu-item`，按钮节点不渲染但用于按钮级判断）。
- **与 PRD"动态路由"的差距**：当前未做组件级动态加载（`import()`），路由表完整注册；差异在文档注明，不影响权限隔离效果——未授权路由即使直接访问也会被守卫拦截。组件级动态加载列为后续增强。

#### 5.5.4 Excel 批量导入设备

- 后端 `POST /api/devices/import`（multer `FileInterceptor('file')`，限 xlsx）+ `GET /api/devices/import-template`（生成模板流下载），见 §5.3.9。
- 前端 `ExcelImportDialog.vue` 组件：拖拽/点击选文件、下载模板、结果表格（成功数、失败数、失败行明细），接入 `DeviceList.vue`"批量导入"按钮。

#### 5.5.5 初始化数据（seed）

- seed 幂等（`upsert` by `path` / `name+parentId` / `username` / `name`）：39 条菜单（三级建模）+ 2 个角色（超级管理员 `menu_ids` 留空、运营人员勾选部分菜单）+ 2 个账号（`admin/admin123`、`operator/operator123`）。
- 运营人员角色用于验证权限隔离：可执行业务全流程的查看与写入，但不可删除任何业务数据，且完全不可见系统管理菜单与接口。

## 6. 实时推送设计

### 6.1 连接认证流程

```
安卓设备                      服务端
   |                            |
   |  1. WebSocket连接
   |  携带 deviceCode 和 timestamp
   |--------------------------->|
   |                            |
   |  2. 验证设备编码合法性
   |  校验时间戳防重放
   |                            |
   |  3. 连接成功
   |<---------------------------|
   |                            |
   |  4. 定期ping心跳 (30s)
   |<-------------------------->|
   |                            |
```

**连接建立参数：**

```
wss://api.adspread.com/socket.io?deviceCode=DEVICE001&timestamp=1716547200000
```

### 6.2 推送事件定义

| 事件名           | 方向        | 说明                                     |
| ---------------- | ----------- | ---------------------------------------- |
| `program_update` | 服务端→设备 | 节目更新通知，设备收到后立即拉取最新节目 |
| `push_ack`       | 设备→服务端 | 设备收到推送后的确认回执                 |
| `device_status`  | 设备→服务端 | 设备状态实时上报                         |
| `ping` / `pong`  | 双向        | 心跳保活                                 |

**推送消息示例：**

```json
{
  "event": "program_update",
  "data": {
    "publishPlanId": 10,
    "programId": 5,
    "reason": "manual_push",
    "timestamp": 1716547200000
  }
}
```

### 6.3 推送实现机制

1. **单设备推送**：按 `deviceCode` 找到对应Socket连接，发送消息
2. **门店批量推送**：查出门店下所有设备编码，批量发送
3. **推送失败重试**：设备不在线时记录待推送队列，设备上线后补发
4. **推送日志**：所有推送操作记录到 push_message_logs 表

### 6.4 推送状态追踪

- **待推送**：已创建推送任务，设备不在线等待上线
- **已推送**：已发送到设备，等待确认回执
- **已送达**：收到设备确认回执，推送成功
- **失败**：推送失败，超过最大重试次数后标记失败

#### 7.4.3 动态路由与权限控制

> **实现说明（v1.3 第二阶段同步）**：实际实现采用**静态路由注册 + 守卫拦截 + 侧边栏按权限动态渲染**。所有路由在构建期静态注册（`meta.permission` 标注所需权限码），`router.beforeEach` 校验目标路由 `meta.permission` 是否在用户权限码集合内，未授权跳转；侧边栏由 `GET /api/auth/menus` 返回的 `menuTree` 递归渲染。未采用组件级动态 `import()` 加载，差异已在 §5.5.3 注明。下方代码示例体现的是"登录后拉取权限菜单用于守卫判断"的思路，`getMenuList()` 用于填充权限码集合而非 `router.addRoute()`。

```typescript
// src/router/permission.ts
import router from '.';
import { useUserStore } from '@/stores/user';
import { usePermissionStore } from '@/stores/permission';

const whiteList = ['/login'];

router.beforeEach(async (to, from, next) => {
  const userStore = useUserStore();
  const permissionStore = usePermissionStore();

  if (userStore.token) {
    if (to.path === '/login') {
      next('/');
    } else {
      if (!permissionStore.menuList.length) {
        // 拉取当前用户可见菜单树，填充权限码集合，用于守卫判断与侧边栏渲染
        await permissionStore.fetchMenus();
        // 校验目标路由 meta.permission 是否在权限码集合内
        if (to.meta?.permission && !permissionStore.hasPermission(to.meta.permission)) {
          next('/dashboard');
        } else {
          next({ ...to, replace: true });
        }
      } else {
        if (to.meta?.permission && !permissionStore.hasPermission(to.meta.permission)) {
          next('/dashboard');
        } else {
          next();
        }
      }
    }
  } else {
    if (whiteList.includes(to.path)) {
      next();
    } else {
      next('/login');
    }
  }
});
```

```typescript
// src/composables/useTable.ts
import { ref, reactive } from 'vue';

export function useTable<T = any>(fetchApi: Function) {
  const loading = ref(false);
  const data = ref<T[]>([]);
  const total = ref(0);

  const queryParams = reactive({
    page: 1,
    pageSize: 20,
  });

  const fetchData = async () => {
    loading.value = true;
    try {
      const res = await fetchApi(queryParams);
      data.value = res.list;
      total.value = res.total;
    } finally {
      loading.value = false;
    }
  };

  const handlePageChange = (page: number) => {
    queryParams.page = page;
    fetchData();
  };

  const handleSizeChange = (size: number) => {
    queryParams.pageSize = size;
    queryParams.page = 1;
    fetchData();
  };

  const refresh = () => {
    queryParams.page = 1;
    fetchData();
  };

  return {
    loading,
    data,
    total,
    queryParams,
    fetchData,
    handlePageChange,
    handleSizeChange,
    refresh,
  };
}
```

```kotlin
fun calculateRegionBounds(
  splitType: String,
  screenWidth: Int,
  screenHeight: Int
): List<RegionBounds> {
  return when (splitType) {
    "1" -> listOf(
      RegionBounds("region1", 0, 0, screenWidth, screenHeight)
    )
    "2" -> if (screenOrientation == PORTRAIT) {
      listOf(
        RegionBounds("region1", 0, 0, screenWidth, screenHeight / 2),
        RegionBounds("region2", 0, screenHeight / 2, screenWidth, screenHeight / 2)
      )
    } else {
      listOf(
        RegionBounds("region1", 0, 0, screenWidth / 2, screenHeight),
```

## 9. 安全设计

### 9.1 认证与授权

#### RBAC权限模型

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  管理员用户  │─────▶│    角色     │─────▶│   菜单权限   │
│  (admins)   │      │   (roles)   │      │   (menus)   │
└─────────────┘      └─────────────┘      └─────────────┘
       1:1                1:N                   树形结构
```

**密码安全：** bcrypt哈希，轮次12，密码长度至少8位，包含大小写字母和数字

#### 接口权限校验（PermissionGuard，v1.3 第二阶段）

全局 `PermissionGuard`（`APP_GUARD`）在 `JwtAuthGuard` 认证之后执行授权，采用**严格兜底**策略——受保护接口（既无 `@Public` 也无 `@AuthenticatedOnly` 又未声明 `@RequirePermission`）一律 403 拒绝，强制所有受保护接口显式声明权限码，避免漏挂导致越权。决策顺序与权限码集合计算见 §5.5.1，接口↔权限码映射见设计规格 §4.4.2。

- 超级管理员角色（名称约定 `超级管理员`）自动放行所有接口。
- 公开接口（登录、`GET /api/device/program`）用 `@Public()` 放行，不经过认证与授权。
- 仅需登录的接口（`/api/auth/me`/`/menus`/`/logout`/`/change-password`）用 `@AuthenticatedOnly()` 放行。
- 业务接口用 `@RequirePermission('xxx:yyy')` 声明所需权限码，与菜单 `permission` 字段一致。

#### JWT Token机制

- Access Token有效期：24小时
- Refresh Token有效期：7天
- Token存储：HttpOnly Cookie + Secure标志
- Token过期后自动刷新机制

## 12. 开发规范

### 12.1 代码规范

- **命名规范**：TypeScript、JavaScript使用camelCase，数据库snake_case，组件PascalCase
- **注释规范**：公共接口、复杂逻辑必须有注释
- **Git提交规范**：`type(scope): message`
  - feat: 新功能
  - fix: 修复bug
  - docs: 文档更新
  - refactor: 重构
  - test: 测试
  - chore: 构建/工具链变动

### 12.2 接口规范

- RESTful风格API
- 统一响应格式
- 分页参数统一：page, pageSize
- 排序参数统一：sortBy, sortOrder
- 筛选参数统一：keyword, status, xxxId等

### 12.3 测试规范

- 单元测试覆盖率 > 80%
- 核心业务逻辑必须有单元测试
- API接口必须有集成测试
- 上线前必须通过回归测试

### 12.4 多语言规范

- 语言key命名规范：`模块名.功能名.具体文案`
- 不允许在代码中硬编码中文字符串
- 新增文案需要同时添加三种语言版本
- 数字、日期、货币格式按语言本地化处理

### 12.5 设计规范

- 遵循设计系统的CSS变量，不允许硬编码颜色值
- 间距使用标准化间距变量（--spacing-xs/sm/md/lg/xl）
- 按钮、表单、弹窗等组件统一使用Element Plus组件
- 业务页面布局保持一致：筛选栏 + 表格 + 分页
