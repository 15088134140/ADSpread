> 📌 **技术栈说明**:
> - **Hermes Agent**: Nous Research 开发的成熟开源 AI 智能体框架（14万+ Stars），核心是"自我进化 + 持久记忆"
> - **Claude Code**: Anthropic 官方的 AI 编码工具，专注 IDE 内代码实现
> - **Superpowers**: 14+ 可组合的质量保障技能集，提供独立视角的审查能力

### Hermes Agent 核心职责

基于 Nous Research 开源的自我进化 AI 智能体框架（14万+ Stars），Hermes Agent 是项目的中枢大脑：

1. **需求理解与任务编排**
   - 自主理解产品需求文档，拆解为多步骤开发任务
   - 定义每个任务的验收标准和技术边界
   - 管理任务依赖关系和执行优先级
   - **持久化记忆**：跨会话记住项目上下文、技术选型、历史决策

2. **自我进化的开发流程**
   - ✨ **技能沉淀**：每次任务完成后自动提炼为可复用的开发技能（Markdown 格式）
   - ✨ **持续优化**：记录踩过的坑、修复过的 Bug，避免重复犯错
   - ✨ **越用越强**：10-20 次相似任务后，执行速度提升 2-3 倍

3. **多 Agent 协调调度**
   - 根据任务类型调度 Claude Code（编码）或 Superpowers（质量审查）
   - 生成隔离的子 Agent 并行处理独立模块
   - 子任务结果汇总和集成协调

4. **全链路工具集成**（内置 47+ 工具）
   - 代码搜索、文件读写、终端执行
   - 网页搜索、文档提取、浏览器自动化
   - 定时任务、跨平台消息推送
   - MCP 协议扩展任意工具能力

5. **质量把关与决策**
   - 代码合并前的最终质量审核
   - 技术选型的决策和风险评估
   - 跨模块架构一致性保障
   - 技术债务追踪和管理

### Claude Code 编码职责

1. **代码实现**
   - 按需求文档和技术设计实现功能
   - 遵循项目编码规范和最佳实践
   - 编写单元测试和集成测试

2. **调试与修复**
   - 快速定位并修复 Bug
   - 性能优化和代码重构
   - 代码质量持续改进

3. **文档编写**
   - 代码注释和 API 文档
   - 技术方案和设计决策记录
   - 使用说明和开发手册

### Superpowers 质量保障技能集

> ✅ **研究验证**: Superpowers 是 14+ 个可组合技能的集合，而非自动化引擎。
> 核心价值是**引入独立 AI 视角**，子 Agent 天然不继承上下文，避免单一 AI 的思维定势和幻觉。

1. **头脑风暴与方案评审** (`brainstorming` 技能)
   - 需求不明确时：多维度发散分析，识别潜在风险
   - 技术选型前：提供多方案对比和 trade-off 分析
   - 架构设计后：独立视角的架构评审，发现 Claude Code 的盲区

2. **代码质量审查** (`code-review` 技能)
   - **正确性审查**: 逻辑验证、边界条件处理、异常流程覆盖
   - **安全性审查**: SQL 注入、XSS、认证授权、数据脱敏
   - **性能审查**: N+1 查询、内存泄漏、算法复杂度、IO 优化
   - **可维护性审查**: 代码可读性、命名规范、注释质量、复杂度
   - **一致性审查**: 是否符合项目规范和架构设计原则

3. **复杂任务的多 Agent 协作** (`subagent-driven-development` 技能)
   - 对于特别复杂的模块，启动子 Agent 并行开发
   - 三角色天然隔离:
     - **Implementer**: 专注功能实现
     - **Reviewer**: 独立审查代码质量
     - **QA**: 编写测试用例并验证功能
   - 避免单一 AI 的思维定势和错误传播

4. **功能验证** (`verify` 技能)
   - 关键功能开发后的手动验证
   - 回归测试场景设计
   - 发布前最终质量把关

## 开发流程规范

基于 Hermes Agent 的自我进化能力，项目采用"AI 优先、人做决策"的开发模式：

---

### 1. 需求分析与方案设计阶段

```
输入: 产品需求文档 / PRD
├─ Hermes Agent: 自主阅读和理解需求
│  ├─ 调用 web_search 检索行业最佳实践
│  ├─ 使用 memory 工具关联历史项目经验
│  └─ [可选] 调用 Superpowers brainstorming 做方案评估
├─ Hermes Agent: 自动拆解为可执行任务清单
│  ├─ 定义每个任务的验收标准
│  └─ 识别技术风险和依赖关系
└─ ✨ Hermes 自动沉淀: "ADSpread 需求分析方法论"技能
输出: 任务分解清单 + 技术方案 + 可复用的分析技能
```

> 💡 Hermes 的核心价值：下次做类似需求时，直接调用沉淀的技能，速度和质量双提升

---

### 2. 开发实施阶段

```
输入: 任务清单
├─ Hermes 智能调度
│  ├─ 简单 CRUD → Claude Code 直接实现
│  ├─ 复杂模块 → 生成隔离的子 Agent 并行开发
│  └─ 核心业务 → 开发完成后自动触发 Superpowers code-review
├─ Claude Code / 子 Agent: 实现功能
│  ├─ 编码实现
│  ├─ 单元测试
│  └─ 本地功能验证
└─ Hermes: 集成协调 + 质量把关
```

**Superpowers Code Review 自动触发条件**（Hermes 内置规则）:
- ✅ 涉及认证、权限、支付等安全敏感逻辑
- ✅ 算法复杂度 O(n²) 以上
- ✅ 跨 3 个以上模块的改动
- ✅ 首次使用新技术/新框架

---

### 3. 集成与验证阶段

```
输入: 各模块代码
├─ Hermes 调用 terminal 工具执行集成测试
├─ [关键功能] 调用 Superpowers verify 做端到端验证
├─ Hermes 自动记录缺陷并分配修复任务
└─ ✨ Hermes 沉淀: "回归测试 Checklist"技能
输出: 通过测试的集成版本
```

---

### 4. 发布阶段

```
输入: 测试通过的版本
├─ Hermes 调用 terminal 执行构建命令
├─ Hermes 调用 Superpowers code-review 做发布前最终检查
├─ Hermes 执行部署脚本（SSH/Docker 等 6 种终端后端）
└─ ✨ Hermes 沉淀: "发布流程最佳实践"技能
输出: 生产环境可用版本
```

> 💡 Hermes Agent 内置终端执行能力，支持 6 种后端执行环境（Local/Docker/SSH/Modal 等）

## 编码规范

### TypeScript 规范

1. **类型安全**
   - 严格启用 `strict: true`
   - 禁止使用 `any` 类型（使用 `unknown` 替代）
   - 所有函数必须定义返回类型

2. **命名规范**
   - 类名: `PascalCase` (如 `UserService`)
   - 函数/变量: `camelCase` (如 `getUserList`)
   - 常量: `UPPER_SNAKE_CASE` (如 `MAX_RETRY_COUNT`)
   - 接口: `PascalCase`，前缀 `I` 可选 (如 `IUser`, `User`)

3. **文件结构**
   - 每个文件只导出一个主要类/函数
   - 文件名与导出实体名一致
   - 相关文件放在同一目录

### NestJS 后端规范

1. **模块化架构**
   - 按业务领域划分模块
   - 每个模块包含: `controller`, `service`, `entity`, `dto`
   - 使用依赖注入实现松耦合

2. **分层原则**
   ```
   Controller (HTTP 层) → Service (业务逻辑层) → Repository (数据访问层)
   ```

3. **错误处理**
   - 使用 NestJS 内置 Exception Filter
   - 统一错误响应格式
   - 业务错误使用自定义异常类

### Vue3 前端规范

1. **组合式 API**
   - 使用 `<script setup lang="ts">`
   - 逻辑按功能组织，而非选项类型
   - 复用逻辑提取为 Composables

2. **组件规范**
   - 组件名使用 `PascalCase`
   - Props 必须定义类型和默认值
   - Emits 必须声明类型

3. **状态管理**
   - 简单状态使用 `ref`/`reactive`
   - 复杂状态使用 Pinia
   - 避免全局状态滥用

## Git 工作流

### 分支策略

```
main (生产环境)
  └── develop (开发环境)
        ├── feature/xxx (功能分支)
        ├── bugfix/xxx  (修复分支)
        └── hotfix/xxx  (紧急修复)
```

### 提交规范

使用 Conventional Commits 格式:

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**Type 类型**:
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 重构
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建/工具相关

**示例**:
```
feat(store): add industry category management
- 新增门店行业分类字段
- 支持按行业分类筛选门店

Closes #123
```

## AI 交互规范

### 给 Claude Code 的指令格式

```markdown
**任务**: [清晰描述任务目标]
**上下文**:
- 相关文件路径
- 依赖关系说明
- 已有的实现基础

**验收标准**:
1. [ ] 功能点 1
2. [ ] 功能点 2
3. [ ] 测试覆盖

**技术约束**:
- 遵循的设计模式
- 性能要求
- 安全要求
```

### 代码 Review 标准

Superpower 进行 Code Review 时检查:

1. **正确性**: 逻辑是否正确，边界条件处理是否完整
2. **安全性**: SQL 注入、XSS、认证授权等安全问题
3. **性能**: N+1 查询、内存泄漏、算法复杂度
4. **可维护性**: 代码可读性、命名、注释、复杂度
5. **一致性**: 是否符合项目规范和架构设计
